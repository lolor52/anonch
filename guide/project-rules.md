# Базовые правила проекта

## Назначение
- Проект развивается как frontend-only веб-приложение по MBTI.
- Backend, Node.js и любые библиотеки для интерфейса не используются.

## Технологические ограничения
- Разрешены только `HTML`, `CSS`, чистый `JavaScript` и `JSON`.
- В репозитории не должно быть `Node.js`-зависимостей, `npm`-конфигов, `package.json`, `package-lock.json`, `node_modules`, сборщиков и внешних JS SDK.
- Для локального запуска и простых проверок допустимы только браузер и стандартная библиотека `Python` без сторонних пакетов.
- Не использовать `React`, `Vue`, `Tailwind`, `Bootstrap` и другие UI-фреймворки.

## Данные и хранение
- Аккаунт, сессия, черновик теста и результат хранятся на стороне клиента.
- Вход в проекте допускается только через `VK ID` и `Yandex ID`.
- Для `VK ID` в браузере использовать актуальный web flow `code + PKCE`, если включён реальный режим без SDK.
- Для `Yandex ID` запрашивать только минимально нужные права и не сохранять OAuth-токен после чтения профиля.
- Не хранить OAuth-токены `Yandex ID` и `VK ID` долговременно в `localStorage`.
- После успешного входа сохранять только локальный профиль пользователя и состояние авторизации.
- Вопросы, группы, описания типов и совместимость хранить в отдельных JSON-файлах.

## Обязательный пользовательский контур
- Тест MBTI содержит 22 вопроса.
- После теста должна быть отдельная страница результата.
- У каждого из 16 типов должна быть своя отдельная страница.
- Должны существовать отдельные разделы со всеми 16 типами и с 4 группами:
  `Analysts`, `Diplomats`, `Sentinels`, `Explorers`.
- Должна быть заглушка: «Скоро будет реализован анонимный чат с подбором подходящих MBTI».

## Требования к реализации
- Архитектура должна быть понятной и расширяемой.
- Все страницы должны быть адаптивными и визуально ровными на мобильных и десктопе.
- Реализацию всегда начинать с анализа текущего состояния проекта.
- Уже сделанное без причины не переписывать.
- Если для реального входа через `VK ID` или `Yandex ID` не хватает настроек, использовать безопасные placeholders, mock-режим и явную инструкцию по подключению реальных значений.
- Для браузерной логики использовать только встроенные Web API, без внешних JS-библиотек и SDK.

## Правила процесса
- После каждого этапа фиксировать:
  - что сделано;
  - какие файлы изменены;
  - как проверить;
  - какие риски или TODO остались.
- Автоматические проверки структуры, данных и ключевых сценариев обязательны. Адаптивность проверять без `Node.js`-инструментов.
